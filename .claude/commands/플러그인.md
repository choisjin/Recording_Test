플러그인 개발, 디버깅, 분석을 수행합니다.

요청: $ARGUMENTS (새로만들기 <이름>|디버그 <이름>|분석 <이름>|목록)

## 플러그인 규칙 (반드시 준수)
- 위치: backend/app/plugins/
- 로딩: module_service.py의 _load_plugin_from_file() — `getattr(mod, 파일명, None)`
- 클래스명 == 파일명 (예: VisionCamera.py → class VisionCamera)
- 외부 DLL: backend/app/modules/에 배치

## 명령별 자동 실행

### "새로만들기 <이름>"
1. 기존 플러그인 분석 (VisionCamera.py, CCIC_BENCH.py, SerialPlugin.py)
2. 공통 패턴 추출:
   - `__init__`: 설정 저장 (host, port 등)
   - `Connect()` → str: 연결
   - `Disconnect()` → str: 해제
3. 스캐폴딩 코드를 생성합니다
4. module_service.py의 list_available_modules()에 등록이 필요하면 추가합니다
5. device_manager.py에 connect_type 추가가 필요하면 추가합니다
6. 프론트엔드 DevicePage.tsx 수정이 필요하면 추가합니다

### "디버그 <이름>"
1. 해당 플러그인 소스 코드를 읽습니다
2. module_service.py에서 로딩 경로를 추적합니다
3. device_manager.py에서 연결/사용 경로를 추적합니다
4. 의존성을 확인합니다 (DLL, 외부 라이브러리)
5. 문제 원인과 해결책을 제시합니다

### "분석 <이름>"
1. 클래스 구조 및 메서드 목록
2. 외부 의존성
3. device_manager 통합 방식
4. 프론트엔드 연동 지점

### "목록"
backend/app/plugins/ 디렉토리의 모든 .py 파일을 나열하고 각각의 클래스명과 Connect 메서드 유무를 보고합니다.

- 기존 플러그인 패턴을 최대한 따릅니다
