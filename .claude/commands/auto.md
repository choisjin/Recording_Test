요청을 분석하여 전체 개발 파이프라인을 자동으로 실행합니다.

요청: $ARGUMENTS

## 파이프라인 실행

### 1단계: 요청 분석 및 파이프라인 선택

$ARGUMENTS를 분석하여 작업 유형을 판단합니다:

- **기능 개발/수정**: 기획 → 조언 → 구현 → 검증 → 완료
- **버그 수정**: 진단 → 구현 → 검증 → 완료
- **플러그인 작업**: 플러그인 분석 → 구현 → 검증 → 완료
- **리팩터링**: 기획 → 구현 → 검증 → 완료

파이프라인을 선택한 후 "📋 파이프라인: [유형]"을 출력하고 즉시 실행합니다.

### 2단계: 기획 (기능 개발/수정/리팩터링 시)

Agent 도구를 사용하여 Explore 에이전트를 실행합니다:
- 요청과 관련된 코드를 탐색합니다
- 변경이 필요한 파일과 영향 범위를 파악합니다
- 기존 코드 패턴과 유틸리티를 확인합니다

결과를 바탕으로 구현 계획을 수립합니다:
- 수정할 파일 목록
- 각 파일의 변경 내용
- 작업 순서

"📋 기획 완료"를 출력하고 다음 단계로 진행합니다.

### 3단계: 조언 (설계 결정이 필요한 경우만)

여러 접근법이 가능한 경우에만 실행합니다:
- 선택지와 트레이드오프를 분석합니다
- 이 프로젝트에 가장 적합한 방법을 선택합니다
- 선택 이유를 간단히 기록합니다

설계 결정이 명확하면 이 단계를 건너뜁니다.

### 4단계: 구현

기획된 계획에 따라 코드를 수정합니다:
- 파일을 읽고 변경사항을 적용합니다
- 기존 코드 패턴과 컨벤션을 따릅니다
- UI 텍스트는 translations.ts를 사용합니다
- 주석은 한국어 허용

### 5단계: 검증

변경된 모든 파일에 대해 자동 검증합니다:

1. **Python 구문**: 변경된 .py 파일마다 `python -c "import ast; ast.parse(open('<file>', encoding='utf-8').read())"`
2. **TypeScript**: 변경된 .ts/.tsx가 있으면 `cd frontend && npx tsc --noEmit`
3. **패턴 정합성**:
   - models/scenario.py 변경 → 프론트엔드 API 호출 확인
   - routers/ 변경 → 프론트엔드 경로/파라미터 확인
   - 새 의존성 → requirements.txt 등록 확인
4. **보안**: .env, credentials, 토큰 하드코딩 여부

검증 실패 시:
- 자동으로 문제를 수정합니다
- 재검증합니다 (최대 2회)
- 2회 실패 시 사용자에게 보고합니다

"✅ 검증 통과" 또는 "❌ 검증 실패 (상세 내용)" 출력

### 6단계: 완료 보고

최종 결과를 요약합니다:

```
## 완료 보고
- 요청: [원래 요청]
- 파이프라인: [실행된 파이프라인 유형]
- 변경 파일: [파일 목록]
- 검증: [PASS/FAIL]
- 다음 단계: `/커밋`으로 커밋하세요
```

## 규칙

- 각 단계 시작 시 단계명을 출력합니다 (예: "🔍 2단계: 기획")
- 구현 전에 사용자 확인을 요청하지 않습니다 — 자동으로 진행합니다
- 커밋은 하지 않습니다 — 완료 후 `/커밋`을 안내합니다
- 검증 실패 시 자동 수정 후 재검증합니다
- 대규모 변경(5개 파일 이상)이면 기획 단계에서 사용자에게 계획을 보여주고 확인을 받습니다
